> | Factor | Weight | Considerations |

### Q2: "How would you implement fairness testing for a credit scoring model?"

**Answer:**
> "I'd implement a comprehensive fairness testing protocol:
>
> **Step 1: Define Protected Attributes**
> - Race, gender, age, disability, religion, national origin
> - Consider proxy variables (e.g., postal code as proxy for race)
>
> **Step 2: Select Fairness Metrics**
> ```
> Metric                    | Definition
> --------------------------|--------------------------------------------
> Demographic Parity        | P(Ŷ=1|A=0) = P(Ŷ=1|A=1)
> Equalized Odds            | Same TPR and FPR across groups
> Predictive Parity         | Same PPV across groups
> Individual Fairness       | Similar individuals get similar outcomes
> ```
>
> **Step 3: Implement Testing Using AIF360**
> ```python
> from aif360.datasets import BinaryLabelDataset
> from aif360.metrics import BinaryLabelDatasetMetric, ClassificationMetric
> from aif360.algorithms.preprocessing import Reweighing
>
> # Load data
> dataset = BinaryLabelDataset(
>     df=credit_data,
>     label_names=['approved'],
>     protected_attribute_names=['gender', 'race']
> )
>
> # Calculate metrics
> metric = BinaryLabelDatasetMetric(
>     dataset,
>     unprivileged_groups=[{'gender': 0}],
>     privileged_groups=[{'gender': 1}]
> )
>
> # Disparate Impact Ratio
> di_ratio = metric.disparate_impact()
> print(f'Disparate Impact: {di_ratio}')
> # Target: 0.8 < DI < 1.25 (Four-Fifths Rule)
>
> # Statistical Parity Difference
> spd = metric.statistical_parity_difference()
> print(f'Statistical Parity Difference: {spd}')
> # Target: |SPD| < 0.1
> ```
>
> **Step 4: Bias Mitigation (if needed)**
> - Pre-processing: Reweighing, sampling
> - In-processing: Adversarial debiasing, constraints
> - Post-processing: Threshold adjustment, reject option classification
>
> **Step 5: Ongoing Monitoring**
> - Monthly fairness dashboards
> - Alerting on metric drift
> - Quarterly deep dives by protected attribute"

### Q3: "Explain SHAP and LIME. When would you use each?"

**Answer:**
> "Both are model-agnostic explainability techniques, but serve different purposes:
>
> **SHAP (SHapley Additive exPlanations):**
> - Based on game theory (Shapley values)
> - Provides consistent, theoretically grounded attributions
> - Shows how each feature contributes to the prediction
>
> ```python
> import shap
>
> # Create explainer
> explainer = shap.TreeExplainer(model)  # or KernelExplainer for any model
>
> # Calculate SHAP values
> shap_values = explainer.shap_values(X_test)
>
> # Visualize
> shap.summary_plot(shap_values, X_test)
> shap.force_plot(explainer.expected_value, shap_values[0], X_test.iloc[0])
> ```
>
> **LIME (Local Interpretable Model-agnostic Explanations):**
> - Fits local linear model around the prediction
> - Fast and intuitive explanations
> - Better for quick, instance-level understanding
>
> ```python
> from lime import lime_tabular
>
> explainer = lime_tabular.LimeTabularExplainer(
>     X_train.values,
>     feature_names=X_train.columns,
>     class_names=['Reject', 'Approve'],
>     mode='classification'
> )
>
> exp = explainer.explain_instance(
>     X_test.iloc[0].values,
>     model.predict_proba,
>     num_features=10
> )
> exp.show_in_notebook()
> ```
>
> **When to Use Each:**
>
> | Scenario | Recommended | Reason |
> |----------|-------------|--------|
> | Regulatory reporting | SHAP | Theoretically grounded, consistent |
> | Customer explanation | LIME | Simpler, faster to generate |
> | Model debugging | SHAP | Global feature importance |
> | Real-time explanation | LIME | Lower computational cost |
> | Audit documentation | SHAP | Better reproducibility |"

### Q4: "What are differential privacy and federated learning?"

**Answer:**
> "Both are privacy-enhancing technologies (PETs) critical for compliant AI:
>
> **Differential Privacy:**
> - Mathematical framework adding calibrated noise to data/outputs
> - Provides provable privacy guarantees (ε-differential privacy)
> - Key concept: Individual's data doesn't significantly affect output
>
> ```python
> # Example: Private mean calculation
> import numpy as np
>
> def private_mean(data, epsilon=1.0, sensitivity=1.0):
>     \"\"\"Calculate mean with differential privacy.\"\"\"
>     true_mean = np.mean(data)
>     noise_scale = sensitivity / epsilon
>     noise = np.random.laplace(0, noise_scale)
>     return true_mean + noise
>
> # Lower epsilon = more privacy, less accuracy
> # Typical range: 0.1 to 10
> ```
>
> **Use Cases:**
> - Training ML models on sensitive data
> - Publishing aggregate statistics
> - Synthetic data generation
>
> **Federated Learning:**
> - Train models across decentralized data without centralizing it
> - Data stays on local devices/servers
> - Only model updates (gradients) are shared
>
> ```
> ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
> │  Client 1   │     │  Client 2   │     │  Client 3   │
> │  (Bank A)   │     │  (Bank B)   │     │  (Bank C)   │
> │  Local Data │     │  Local Data │     │  Local Data │
> └──────┬──────┘     └──────┬──────┘     └──────┬──────┘
>        │                   │                   │
>        │    Model Updates  │                   │
>        └───────────────────┼───────────────────┘
>                            │
>                     ┌──────▼──────┐
>                     │   Central   │
>                     │   Server    │
>                     │  Aggregate  │
>                     └─────────────┘
> ```
>
> **Use Cases:**
> - Cross-institutional AML models
> - Healthcare AI without sharing patient data
> - Mobile keyboard prediction
>
> **Compliance Relevance:**
> - Enables AI development while maintaining data localization
> - Supports PDPA, PIPL cross-border transfer restrictions
> - Demonstrates privacy-by-design approach to regulators"

# SCENARIO-BASED QUESTIONS

### Scenario 1: GenAI Deployment Risk

**Question:** "The wealth management business wants to deploy a GenAI chatbot for investment advice within 3 months. What are your concerns and how would you approach this?"

**Answer:**
> "This is a high-risk deployment requiring careful evaluation:
>
> **Key Concerns:**
> 1. **Hallucination Risk:** GenAI may provide inaccurate investment advice
> 2. **Suitability:** May not properly assess customer risk profiles
> 3. **Regulatory:** MAS Notice 648 (Financial Advisers), HKMA GenAI Guidelines
> 4. **Fair Dealing:** Ensuring advice is in customer's best interest
> 5. **Explainability:** Customers entitled to understand basis of advice
>
> **My Approach:**
>
> *Phase 1: Risk Assessment (Week 1-2)*
> - Classify as high-risk AI (customer-facing, financial decisions)
> - Full privacy impact assessment
> - Map regulatory requirements
>
> *Phase 2: Control Design (Week 3-6)*
> - RAG architecture with approved content only
> - Human-in-the-loop for all investment recommendations
> - Clear disclosure that AI is assisting (not autonomous)
> - Customer consent mechanism
> - Explainability for all recommendations
>
> *Phase 3: Testing (Week 7-10)*
> - Comprehensive fairness testing by customer segment
> - Red-team testing for hallucination/manipulation
> - User acceptance testing with compliance scenarios
>
> *Phase 4: Deployment (Week 11-12)*
> - Phased rollout (internal → pilot → production)
> - Enhanced monitoring for first 90 days
> - Clear escalation paths
>
> **Timeline Assessment:**
> 3 months is aggressive but achievable if we:
> - Start risk assessment immediately
> - Run activities in parallel where possible
> - Have dedicated resources from Technology and Business
> - Accept that scope may be limited initially (e.g., specific products only)"
